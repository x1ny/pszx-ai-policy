# shadcn/ui 语义槽位速查表

shadcn 项目的颜色体系定义在 `src/index.css` 的 `@theme inline` + `:root`/`.dark` 里（Tailwind v4 CSS-first 配置）。做品牌色替换前，先搞清楚这些槽位分别管什么、彼此的关联关系，避免"只改一个变量,其他地方违和"。

## 槽位一览

| 槽位 | 典型用途 | 替换品牌色时要不要联动改 |
|---|---|---|
| `--background` / `--foreground` | 页面底色 / 正文文字色 | 一般不用动，除非整体调性要变 |
| `--card` / `--card-foreground` | 卡片底色 / 卡片内文字 | 一般不用动 |
| `--popover` / `--popover-foreground` | 弹出层（下拉、tooltip） | 一般不用动 |
| `--primary` / `--primary-foreground` | 品牌主色 / 主色上的文字色 | **必改**，这是品牌色替换的核心槽位 |
| `--secondary` / `--secondary-foreground` | 次级强调（次要按钮等） | **要检查**，如果次级色是从主色派生的浅色/邻近色，需要跟着调 |
| `--accent` / `--accent-foreground` | 悬浮态、选中态背景 | **要检查**，很多组件的 hover 背景直接引用这个槽位，如果不跟着改，主色是新蓝但 hover 背景还是旧色系，会很违和 |
| `--muted` / `--muted-foreground` | 中性弱化背景/文字（次要信息） | 通常保持中性灰即可，但如果原型的"弱化背景"带一点品牌色调（例如原型的浅蓝背景 `#eef6ff`），可以让 muted 稍微偏向品牌色相 |
| `--destructive` | 危险操作、错误态 | 一般保持红色系不变，除非原型的"危险色"明显不同 |
| `--border` / `--input` / `--ring` | 边框 / 输入框边框 / focus 环 | `--ring` 常常直接等于 `--primary` 或其变体，替换主色时顺带检查 |
| `--chart-1` ~ `--chart-5` | 图表配色序列 | 如果原型的图表/图例有专属配色（如匹配度高中低），大概率不该塞进 chart-1~5 这种"图表分类色序列"，而应该走下面的"新增业务语义槽位" |
| `--radius` 及派生的 `radius-sm/md/lg/xl` | 圆角刻度 | 通常直接复用，不需要因为品牌色替换而改动 |

## 替换主色的操作顺序

1. 确定原型的品牌主色（primitive 值，如 `#1677ff`），写入 `--primary`，同时确认 `--primary-foreground` 在新主色上对比度是否依然够（浅色主色可能需要把 foreground 从白色换成深色文字）。
2. 检查 `--accent`：很多 hover/选中态背景直接读这个槽位。如果原型的 hover 背景是主色的浅色版本（如 `#eef6ff`），把 `--accent` 调整为主色对应的浅色调，而不是留着 shadcn 默认的中性浅灰紫。
3. 检查 `--secondary`：如果原型里"次级按钮/次级强调"用的是主色的另一种浓淡，同步调整；如果原型里次级色其实是独立的中性灰（不跟随主色），可以不用动。
4. 检查 `--ring`：focus 环颜色，一般让它继续跟随 `--primary`（大多数 shadcn 默认配置本来就是 `--ring: var(--primary)` 或近似值），不需要额外操作，但要确认没有被写死成别的颜色。
5. `--muted`、`--destructive`、`--border`、`--radius`、字号刻度：默认不动，除非原型在这些位置有明显、跨多处一致的独立设计意图。

## 何时新增自定义业务语义槽位

只有当：
- shadcn 默认槽位里找不到语义对应的，**并且**
- 这个语义是项目独有的业务概念（不是"又一个装饰色"），**并且**
- 满足 SKILL.md 里的三条经验法则（复用性/易变性/语义独立性）

才在 `@theme` 里新增一组，命名按业务角色（如 `--match-high` / `--match-mid` / `--match-low`），不要塞进 `chart-1~5` 或者硬套一个语义不完全贴合的现有槽位（比如把"中匹配"的橙色硬叫成 `--warning`，如果它在原型里的实际含义不是"警告"而是"匹配度中等"，两者语义不等价，硬塞进 `warning` 会在其他真正表示"警告"的场景造成误导）。

## 一个常见反例

原型里经常会看到类似这样的自定义变量块（这是设计者自己收敛出的调色板）：

```css
:root {
    --primary: #1677ff;
    --primary-2: #22c7ff;
    --green: #15b881;
    --purple: #7c3aed;
    --orange: #f59e0b;
}
```

`--primary` 直接对应 shadcn 的 `--primary` 槽位，可以直接映射。但 `--green`/`--purple`/`--orange` 这种按颜色命名的变量，往往只是原型作者写 CSS 时图方便建的别名，不代表它们真的是需要全局复用的语义 token——去原型里搜一下这几个变量实际被引用的地方，如果只在一两个孤立的装饰场景使用（比如某个多阶段卡片各阶段的强调色），应该按 SKILL.md 的三层模型下沉为 component-local，而不是照抄进 `@theme`。

# 完整案例：从一份内部管理后台原型提取 Token

背景：团队拿到三份 HTML 原型（首页看板、详情页 A、详情页 B），要在一个已经用 Tailwind v4 + shadcn/ui 搭好基础布局的项目里还原视觉。首页原型是纯手写 CSS，另外两份用了 Tailwind CDN + 补充样式，三份文件风格并不统一——这是原型交付里非常常见的情况，不是特例。

## 第 1 步：识别最终设计意图，而不是照单全收

首页原型的 CSS 里能看到一连串按迭代顺序追加的注释块（类似"第 N 轮优化"），每一轮都在文件末尾加一批新规则去覆盖前面的效果。这说明：

- 文件里同一个"品牌蓝"语义，先后出现了两个不同的具体数值（旧版本一个、新版本一个）。**不能把两个都当 token 候选**，要看最终生效的是哪个——通常是最后一轮追加、优先级最高（常常还带 `!important`）的那个。
- 幸运的是，最后一轮优化里，原型作者自己写出了一段结构清晰的变量声明：

  ```css
  :root {
      --primary: #1677ff;
      --primary-2: #22c7ff;
      --primary-soft: #eef6ff;
      --green: #15b881;
      --purple: #7c3aed;
      --orange: #f59e0b;
      --text-main: #172033;
      --text-sub: #6b7280;
      --line: #e7edf6;
      --card-shadow: 0 12px 30px rgba(31, 89, 171, 0.08);
  }
  ```

  这是作者自己收敛出的设计意图，应该以它为 token 提取的起点，而不是从整份文件散落的 hex 值里自己瞎猜。

## 第 2 步：按语义聚类，逐个过三条经验法则

| 候选 | 复用性（3处+？） | 易变性（会被整体调？） | 语义独立性（有独立业务含义？） | 结论 |
|---|---|---|---|---|
| `--primary #1677ff` | 是，全站按钮/链接/高亮到处用 | 是，品牌色随时可能调整 | 是，"品牌主色"是独立概念 | **提升为 semantic token**，映射到 shadcn `--primary` |
| 匹配度高/中/低配色（分散在表格、卡片、图例里，实际颜色是绿/橙/蓝或灰） | 是，至少 4 处不同组件复用同一套语义 | 是，产品可能调整这套颜色的具体色值 | 是，"匹配度高中低"是核心业务概念，颜色只是当前的视觉实现 | **提升为 semantic token**，新增业务槽位 `--match-high/mid/low`（shadcn 默认没有对应槽位） |
| `--purple #7c3aed` / `--orange #f59e0b` | 否，实际搜索后发现只在"培育计划四阶段"卡片里，每阶段各用一次 | 否，没有迹象表明会被要求整体调整 | 否，只是"第几个阶段"的视觉区分，没有脱离颜色本身的独立业务含义 | **下沉为 component-local**，作为培育阶段卡片组件内部的一个强调色数组，不进全局 token |
| 各处圆角（4/6/8/10/12/14/16/18px 混用） | 是，圆角这个"设计维度"当然到处用 | 是 | 否，圆角本身没有业务语义，纯粹是尺寸刻度 | 不需要新建自定义 token，**直接复用 shadcn 已有的 `--radius` 派生刻度**（sm/md/lg/xl），把混乱的数值往这几档上收敛 |

注意 `--purple`/`--orange` 这两个例子：它们在原型里**确实被声明成了 CSS 变量**，如果不做这层过滤，很容易顺手把它们也搬进全局 `@theme`。命名上它们是"按颜色命名"（green/purple/orange），这本身就是一个信号——提醒你回去检查它们是否真的有跨组件复用的业务语义。检查后发现没有，就应该下沉，而不是因为"原型里写了变量"就默认它该进全局配置。

## 第 3 步：映射到 shadcn 槽位

- `--primary` → 直接替换 shadcn 的 `primary` 槽位；同时按 [shadcn-slots.md](shadcn-slots.md) 的流程检查 `accent`（悬浮态背景，原型的浅蓝 `--primary-soft: #eef6ff` 正好可以喂给它）和 `ring`（focus 环通常跟随 primary，一般不用额外处理）。
- `--match-high/mid/low` → shadcn 默认主题没有这组语义，确认三条经验法则都满足后新增，不要硬塞进 `destructive`/`chart-1~5` 这类语义不完全贴合的现有槽位。
- 圆角/阴影/字号 → 沿用 shadcn 已有刻度，原型里散落的具体数值只是拿来对照"大致该收敛到哪一档"，不需要逐个照抄。

## 第 4 步：交付决策表

| Token 名 | 层级 | 值 | 依据 |
|---|---|---|---|
| `--primary` | semantic（替换 shadcn 槽位） | `#1677ff` | 全站高频复用的品牌主色，来自原型最后一轮迭代的收敛值 |
| `--accent` | semantic（联动调整） | 基于 `#eef6ff` 调整 | 主色替换后，悬浮态背景需要跟着调整色相，否则和新主色违和 |
| `--match-high` / `--match-mid` / `--match-low` | semantic（新增槽位） | 绿 / 橙 / 中性灰 | 项目独有业务语义，shadcn 无对应槽位，4 处以上跨组件复用 |
| 圆角/阴影 | 复用 shadcn 已有 | 不新增 | 原型数值只是参考，收敛进已有 `--radius` 刻度即可 |
| 培育阶段强调色 | component-local | `['emerald', 'orange', 'violet', 'cyan']` | 只在单一组件内部按 index 使用，不满足复用性/语义独立性，不进全局 token |

这张表交付给用户确认之后，再动手改 `index.css` 的 `@theme` 和对应组件代码——先对齐判断依据，再落地代码，避免"方案没谈拢就先把代码写好了"的返工。

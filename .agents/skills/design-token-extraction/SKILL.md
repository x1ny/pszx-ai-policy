# Design Token Extraction

## 这个 skill 解决什么问题

拿到一份 HTML 原型或设计稿，照抄一遍样式很容易；难的是提炼出一套**将来改起来不痛苦**的 token 规范。这个过程有两个方向相反但同样常见的失败：

- **欠抽象**：组件里到处写死颜色值/圆角/间距，改一次品牌色要全仓库搜替换，原型里"同一个蓝色"出现 `#1890ff` 和 `#1677ff` 两个数值混用而没人发现，就是这个失败的典型后果。
- **过度抽象**：看到什么值都建一个全局变量，token 文件膨胀到没人敢乱动（不知道还有谁在引用），改一个只用一次的装饰色也要去翻全局配置——这比不用 token 更麻烦。

这个 skill 的核心不是"教你写 CSS 变量"，而是教你**判断一个值到底值不值得提升为 token，提升到哪一层**。把这个判断做对，token 系统才是净收益而不是新负担。

## 核心心法：Token 对应"设计决策"，不对应"用到的每个值"

在动手之前先建立这个前提：token 不是给"原型里出现过的值"建目录，而是给"未来可能需要统一调整的设计决策"建一个可被引用的名字。凡是只在一处使用、大概率不会被要求"整体调整"的值，直接写字面量（或用 Tailwind 默认调色板里现成的值）反而更清楚，不需要包一层抽象。

## 三层 Token 模型

处理任何一个候选值（颜色、圆角、间距、阴影）之前，先判断它属于哪一层：

| 层级 | 定义 | 例子 | 存放位置 |
|---|---|---|---|
| **Primitive（原始值）** | 一个具体的数值本身，不带业务含义 | `#1677ff`、`8px`、`0 12px 30px rgba(31,89,171,.08)` | 只有"品牌相关、且会被语义层引用"的才需要单独定义；其余优先直接用 Tailwind 默认调色板（`blue-600`、`emerald-500`……），不必重新发明一套色板 |
| **Semantic（语义别名）** | 描述"这个值在系统里扮演什么角色"，引用 primitive 或直接定值 | `--primary`、`--match-high`、`--danger` | 全局 `@theme`，组件代码只允许引用这一层，不允许直接引用 primitive |
| **Component-local（组件内局部配置）** | 只在一个组件内部按场景/index 切换的样式，不具备跨组件复用价值 | 一个多阶段时间轴组件里每个阶段各一个强调色 | 组件文件内部的一个数组/映射（如 `const stageAccent = ['emerald', 'orange', 'violet', 'cyan']`），不进全局 CSS 变量 |

**组件永远只引用 semantic 层**，不直接写 primitive 值，也不直接写死 hex/px。这是"以后要整体调整品牌色"这句话成立的前提——只改语义层的映射，组件代码完全不用动。

## 判断"要不要提升为全局 token"的三条经验法则

遇到原型里任何一个候选值，依次问这三个问题。**三条都满足**才值得放进全局 semantic token；只满足一两条的，大概率该留在 component-local 层，或者干脆用字面量。

1. **复用性**：这个语义是否在 3 处以上、不同组件里重复出现？（"匹配度-高"会出现在表格、卡片、图例、tag 里，够格；某个卡片装饰用的渐变色只出现一次，不够格）
2. **易变性**：这个值未来有没有可能被整体调整？（品牌主色几乎必然会被要求调整，值得 token 化；一次性装饰色基本不会有人说"把所有这个色调统一改掉"，不值得）
3. **语义独立性**：这个值是否代表一个独立于"具体颜色/数值"的业务概念？（"匹配度高中低"是业务概念，颜色只是这个概念当前的视觉实现，两者应该解耦；反之如果一个值纯粹是视觉选择，没有业务含义可以脱离颜色单独描述，通常不需要语义化）

## 命名原则：按用途命名，不按外观命名

语义 token 的名字应该回答"这是干什么用的"，不是"这是什么颜色"。

- ✅ `--match-high` / `--match-mid` / `--match-low`、`--danger`、`--primary`
- ❌ `--green`、`--purple`、`--orange`

**如果发现自己在按颜色本身命名一个"全局" token，这本身就是一个信号，说明这个值其实还停留在 primitive 层，被误当成语义层提前放进了全局配置。** 遇到这种命名，先回去用三条经验法则重新过一遍：它到底有没有独立于"这个颜色"的业务含义？如果没有，它应该下沉为 component-local，而不是留在全局起一个按颜色命名的变量。

## 与 shadcn/ui 协作的规则

项目如果已经用 shadcn/ui，**不要重新发明一套版式系统**，shadcn 的 spacing/typography/radius/shadow 骨架已经是经过验证的克制体系，直接复用。真正需要处理的通常只是色相层：

1. **先盘点 shadcn 已有的语义槽位**，缺什么补什么，不要重复造轮子：
   `primary` / `primary-foreground`、`secondary` / `secondary-foreground`、`accent` / `accent-foreground`、`muted` / `muted-foreground`、`destructive`、`border` / `input` / `ring`、`card` / `popover`、`chart-1`~`chart-5`、以及 `--radius` 派生出的 `radius-sm/md/lg/xl`。
2. **替换品牌色不是只改一行 `--primary`**。`secondary`/`accent`/`muted` 这几个槽位的色相通常是跟主色相关联设计的（比如浅色背景、描边色），只改 primary 会出现"主色是新的，但浅色背景/描边还是旧色相"的违和感。换品牌色时要联动检查这几个槽位是否协调。
3. **只有 shadcn 默认槽位里确实没有、且是项目独有业务语义的，才新增自定义 token 组**。例如"匹配度高中低"这种业务专属的三态语义，shadcn 通用主题里不存在对应槽位，这时才新增 `--match-high/mid/low`，而不是把这类业务色硬塞进 `success`/`warning` 之类通用槽位（语义不完全等价时，硬塞进去会在别的场景造成误导）。
4. **颜色优先用 Tailwind 默认调色板**，只有品牌色本身不在默认色板里（比如原型给的是设计师自定义的一个具体蓝色）时，才需要单独定义一个 primitive token。

需要具体槽位速查表和替换步骤时读 [references/shadcn-slots.md](references/shadcn-slots.md)。

## 从原型提取 Token 的实操流程

### 第 1 步：通读原型，识别"最终设计意图"，而不是简单收集所有值

原型 CSS 往往不是一次写成的，会有多轮迭代痕迹（版本注释、文件末尾追加的覆盖规则、新旧数值混用）。**不要把历史上出现过的所有颜色/数值都当成候选**，要先判断"哪一版才是作者最后想要的效果"：

- 看是否有明确的迭代注释（比如"本轮优化"之类的标记），越靠后的通常越接近最终意图。
- 如果同一个语义（比如"品牌蓝"）在文件里有多个不同的具体数值，检查实际渲染生效的是哪一个（后声明的规则、`!important`、更高优先级选择器通常会覆盖前面的），以生效的那个为准，而不是把所有历史数值都当 token 候选去重。
- 如果原型作者自己在某一轮里已经写出了一套结构清晰的 CSS 变量声明（`:root { --xxx: ... }`），优先以这套变量作为 token 来源的起点——这通常就是作者自己收敛出的设计意图，比自己重新从散落的 hex 值里瞎猜要可靠。

### 第 2 步：按语义聚类，而不是按数值去重

把原型里的颜色/圆角/间距按"它在界面里的作用"分组，而不是按具体数值。同一个语义分组里如果有多个相近但不同的数值（如 `#1890ff` 和 `#1677ff` 都在表达"品牌蓝"），说明这是需要清理的历史遗留，不是两个不同的语义。

### 第 3 步：逐个候选值过三条经验法则

对每个语义分组，跑一遍"复用性/易变性/语义独立性"三条问题，决定它属于 primitive / semantic / component-local 哪一层（见上面的三层模型）。**不要因为原型里声明了一个 CSS 变量就默认它该进全局 token**——原型作者建变量可能只是图自己写着方便，不代表这个值真的需要跨组件复用。一次性装饰色（比如某个多阶段卡片每个阶段一个强调色）即使原型里写成了变量，也应该下沉为组件局部配置。

### 第 4 步：把语义 token 映射到 shadcn 槽位

对通过筛选、确认要做成 semantic token 的值：
- 先检查是否对应 shadcn 已有槽位（品牌色 → `primary`；危险/警示 → `destructive`；中性背景 → `muted`），能复用的不要新建。
- 换色时联动检查关联槽位（见上面"与 shadcn 协作的规则"第 2 条）。
- 只有确认是项目独有业务语义、shadcn 没有对应槽位时，才新增一组自定义 token，并按"命名原则"用业务角色命名。

### 第 5 步：交付一张 Token 决策表，而不是一坨 CSS

最终产出不应该是直接甩一段 `@theme` 代码，而是先给出一张可读的决策表，让使用者能看懂"为什么这样定"，再落地成代码：

| Token 名 | 层级 | 值/来源 | 决策依据 |
|---|---|---|---|
| `--primary` | semantic | 原型 `#1677ff`（覆盖 shadcn 默认） | 品牌主色，跨全站高频复用，属于会被整体调整的设计决策 |
| `--match-high/mid/low` | semantic（新增槽位） | 原型表格/卡片/图例统一色 | shadcn 无对应槽位，是项目独有业务语义，3处以上复用 |
| 培育阶段强调色 | component-local | 原型 4 个阶段各一色 | 只在一个组件内部按 index 使用，不满足复用性/语义独立性 |

决策表定下来之后，再写入 `@theme` 对应的 CSS 变量，或组件内部的局部配置。

需要一个完整的端到端案例时读 [references/worked-example.md](references/worked-example.md)。

## 反模式警示

遇到以下情况，直接指出问题，不要顺着继续做：

- **Token 无限增殖**：每遇到一个新值就建一个新 token，最后 token 文件比组件代码还难维护。判断标准永远是三条经验法则，不是"原型里有就抄"。
- **用 `!important` 层层覆盖代替修改语义层**：如果发现自己在为了改一个视觉效果不断在文件末尾追加新的覆盖规则，而不是回去改语义 token 的定义，这就是 token 系统失效的信号——说明当初要么没有建对应的语义层，要么组件直接写死了原始值导致没法通过改一处生效。
- **每个页面/组件各自发明间距、圆角数值**：多个页面里出现"看起来差不多但数值不同"的圆角/间距（比如 6px/8px/10px/12px 分不清是谁的设计决策），这说明缺一个统一刻度，应该收敛进已有的 spacing/radius 体系，而不是让每个页面自成一派。

## 交付形式

完成分析后，按以下顺序呈现给用户，而不是直接改代码：
1. 先给出 Token 决策表（层级 + 值 + 依据），让用户能审阅"为什么这样分层"。
2. 明确列出：复用 shadcn 已有槽位的部分 / 新增的语义槽位 / 下沉为 component-local 的部分。
3. 得到确认后，再落地为 Tailwind v4 `@theme` 的 CSS 变量定义，以及需要下沉的组件局部配置代码。
